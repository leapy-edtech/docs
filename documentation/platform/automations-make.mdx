## Contexto de Produto

Make (anteriormente Integromat) é a plataforma de automação no-code da Leapy. Hoje cobre
três áreas principais:

1. **Pipeline de presenças Zoom** — coleta + reconciliação diária.
2. **Comunicações por status** — notificações automáticas quando uma vaga muda de status ou
   um jovem novo é criado.
3. **Agregação de dashboards** — atualização periódica de tabelas analíticas.

Outros caminhos (comunicações de régua, follow-up WhatsApp, alinhamento de presença) são
implementados via **Inngest functions + hooks Directus**, não Make. Ver
[Eventos e Jobs Inngest](/documentation/platform/events-jobs-inngest) e
[Régua de Comunicação e Follow-up](/documentation/platform/communications-fup).

## Conceitos

### Cenários

Cenários Make são workflows automatizados compostos por módulos encadeados:

```mermaid
flowchart LR
    A[Trigger] --> B[Ação 1]
    B --> C[Filtro]
    C --> D[Ação 2]
    D --> E[Ação Final]
```

| Tipo de Módulo | Descrição |
|---|---|
| **Trigger** | Inicia o cenário (webhook, schedule, polling) |
| **Ação** | Operação atômica (consultar API, transformar dados, enviar email) |
| **Filtro** | Conditional — passa adiante ou não |
| **Router** | Divide o fluxo em múltiplos caminhos |
| **Aggregator** | Agrupa itens de uma iteração |

### Acesso ao painel

UI: [us1.make.com/scenarios](https://us1.make.com/71159/scenarios). Cada cenário tem
histórico de execuções, blueprint editável, logs por módulo.

## Integração com Inngest

Make e Inngest convivem na Leapy — Make dispara crons e gerencia integrações externas;
Inngest executa jobs assíncronos do código. Os dois se comunicam via eventos:

### Make → Inngest (mais comum)

Cenários Make como `Alinhar presenças` chamam o endpoint Inngest pra disparar um evento:

```typescript
// No blueprint Make, módulo HTTP > "Make a request":
// POST <INNGEST_INGEST_URL>
// Body:
{
  "name": "backoffice/alinhar-presencas.send",
  "data": { "ids": [101, 102, 103] }
}
```

### Inngest → Make (raro)

Inngest pode chamar webhooks Make via fetch em `step.run`, mas hoje a Leapy usa pouco essa
direção — preferencialmente os jobs Inngest fazem o trabalho localmente.

## Diagnóstico Operacional

Consultas comuns durante investigação de problemas, via UI Make ou via API:

| Pergunta | Onde checar |
|---|---|
| "Que cenários existem com nome contendo X?" | UI: filtro de busca em [us1.make.com/scenarios](https://us1.make.com/71159/scenarios). API: `GET /scenarios?teamId=<id>&name_contains=X` |
| "Esse cenário rodou em data Y?" | UI: `us1.make.com/scenarios/<id>/history`. API: `GET /scenarios/<id>/executions?from=<data>` |
| "Detalhes do erro da execução Z?" | UI: clicar na execução no history. API: `GET /scenarios/<id>/executions/<execution_id>` |

> Em investigações de campo, a **leitura** dos cenários é segura. Mutações (rodar manualmente,
> editar blueprint, ativar/desativar) afetam produção em tempo real e devem passar pelo time
> tech responsável pelo cenário, com backup do blueprint antes de qualquer alteração.

## Gestão de Cenários

### Versionamento

Cenários ativos hoje **não têm versionamento formal** — o blueprint é editável live na UI. Para
mudanças críticas: exportar blueprint via "Export Blueprint" antes de alterar, manter cópia
local, e usar "Replace Blueprint" pra rollback se necessário.

### Monitoramento

- **Histórico de execuções:** `us1.make.com/scenarios/<id>/history` — últimas 100 execuções.
- **Status do cenário:** ativo / inativo / pausado. Status `error` em ≥3 execuções
  consecutivas pausa o cenário automaticamente.
- **Operations consumption:** painel mostra operações usadas vs disponíveis no plano.

### Ambientes

A conta atual cobre apenas **produção**. Cenários com `[HMG]` ou `[POC]` no nome são restos
históricos de homologação — verificar se ativos antes de tocar.

## Boas Práticas

1. **Nomeação clara**: prefixar com domínio (`[Presença]`, `[Pulsos]`, `[Vagas]`) facilita
   filtro via `make_scenarios_list({name_contains: ...})`.
2. **Filtro inicial**: cenários com webhook devem ter filtro logo após o trigger pra ignorar
   payloads irrelevantes (reduz operations consumption).
3. **Error handling**: usar módulos `Error Handler` (resume / rollback / commit) — não deixar
   o cenário pausar silenciosamente em erro transitório.
4. **Idempotência**: cenários reexecutados não devem duplicar efeito colateral (usar
   `update` em vez de `create` quando aplicável).
5. **Documentar IDs em referência cruzada**: incluir o `scenario_id` Make em comentário no
   código Inngest correspondente, e vice-versa.

<Warning>
  Alterações em cenários ativos afetam **produção imediatamente** — não há ambiente sandbox
  isolado. Sempre exportar blueprint antes de mexer.
</Warning>
