## Contexto de Produto

Inngest é a plataforma de orquestração de eventos e jobs assíncronos da Leapy, hospedando
50+ functions distribuídas em domínios (presenças, follow-up, régua de comunicação, pulsos,
matrículas, Moodle, vagas). O código fica em [`leapy-edtech/backoffice-inngest-functions`](https://github.com/leapy-edtech/backoffice-inngest-functions/tree/main/src/inngest/functions).

Em complemento, [Automações Make](/documentation/platform/automations-make) cuida de crons
e integrações externas. Inngest cuida do trabalho do código (envio de emails, agregação de
dados, sync de progresso).

## Conceitos

### Eventos

Eventos são disparados quando algo acontece — direto pelo código ou via webhook externo.
Cada evento tem um nome canônico e um payload tipado:

```typescript
await inngest.send({
  name: "backoffice/alinhar-presencas.send",
  data: {
    ids: [101, 102, 103],
    chunk_size: 10,
  },
});
```

### Functions

Functions reagem a eventos com lógica de negócio. Cada function tem `id` único, declara o
evento que escuta e retorna um resultado:

```typescript
export const sendReguaComunicacaoJovens = inngest.createFunction(
  { id: "send-regua-comunicacao-jovens" },
  { event: "backoffice/regua-comunicacao/jovens.send" },
  async ({ event, step }) => {
    for (const jovem of event.data.data) {
      await step.run("send-lembrete-jovens", async () => {
        return await sendEmail(jovem.jovem_email_corporativo, event.data.evento);
      });
    }
  },
);
```

### Steps

Steps são as unidades atômicas de retry. Cada `step.run` é uma transação — em caso de erro,
Inngest re-tenta apenas o step falhado, mantendo os anteriores.

### Scheduled Functions

Functions com `cron` rodam por agendamento. A Leapy também usa **hooks Directus com schedule**
pra disparar eventos Inngest indiretamente — alguns crons vivem no Directus, outros no
Inngest direto.

## Retry e Error Handling

Por default, Inngest re-tenta steps falhados até **3 vezes** com backoff exponencial.
Functions críticas podem override:

```typescript
inngest.createFunction(
  {
    id: "process-webhook",
    retries: 5,
    onFailure: async ({ event, error }) => {
      await notifyTeam(error);
    },
  },
  { event: "webhook.received" },
  async ({ event }) => {
    // …
  },
);
```

## Diagnóstico Operacional

Consultas comuns via dashboard ou API REST do Inngest:

| Pergunta | Onde checar |
|---|---|
| "Esse job rodou em data X?" | Dashboard: filtro por function + janela. API: `GET /v1/events?name=<event>&triggered_after=<data>` + `GET /v1/events/{id}/runs` |
| "Detalhes do erro do run Y?" | Dashboard: clique no run. API: `GET /v1/runs/<run_id>` (atenção: este endpoint pode estar instável — preferir o caminho via `/events/{id}/runs`). |

Dashboard: [app.inngest.com](https://app.inngest.com).

## Dashboard Inngest

Permite:
- Visualizar eventos em tempo real.
- Monitorar execução de functions (success rate, latência, erros).
- Ver logs detalhados por run.
- Reprocessar eventos com falha (botão "Replay").
- Cancelar execuções em andamento.

## Boas Práticas

1. **Idempotência**: functions devem produzir o mesmo efeito ao re-executar com mesmo
   payload (Inngest pode replay).
2. **Steps granulares**: dividir em `step.run` pequenos pra retry eficiente — não fazer
   tudo num único step.
3. **Timeouts em operações externas**: `step.run` com chamada HTTP deve ter timeout
   explícito.
4. **Logs estruturados**: `console.log(JSON.stringify({ event_id, jovem_id, ... }))` —
   facilita filtro no dashboard.
5. **Alertas para falhas críticas**: usar `onFailure` em functions críticas (envio de email
   transacional, sync de Moodle).
6. **Naming convention para eventos**: `<dominio>/<acao>` (ex: `backoffice/alinhar-presencas.send`).
   Domínio em snake-kebab, ação no infinitivo + sufixo `.send` / `.update` / `.created`.
