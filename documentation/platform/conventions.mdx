### Contrato de Dados (DTO)

DTO (*Data Transfer Object*) é o contrato de dados que o frontend consome.

- Não precisa refletir 1:1 o schema do banco ou do Directus.
- Pode combinar campos de múltiplas fontes.
- Pode incluir campos derivados/calculados.
- Deve ser estável e orientado ao caso de uso da tela/API consumidora.

Exemplo de DTO para lista de talentos:

```json
{
  "id": "uuid",
  "nome": "Maria Silva",
  "email": "maria@empresa.com",
  "faltasCurso": 2,
  "ultimoPulso": "2026-02-10",
  "nomeLider": "João Santos",
  "skills": ["React", "Node.js"]
}
```

### Matriz de Decisão: Onde montar o DTO

| Cenário | Padrão recomendado |
|---------|--------------------|
| CRUD e leitura simples de 1 collection | Usar endpoint padrão do Directus (`/items/...`) |
| Composição de múltiplas fontes no servidor do app | Usar serviço no Next.js (BFF) |
| Tela client-side precisa `fetch` HTTP | Expor endpoint no Next (`/api/...`) chamando o mesmo serviço |
| Múltiplos consumidores + dados/lógica 100% Directus + necessidade forte de performance/governança | Endpoint customizado no Directus |
| **Escrita transacional em múltiplas tabelas (atomicidade obrigatória)** | **Endpoint customizado no Directus** |

### Critérios para endpoint customizado no Directus

Crie endpoint customizado no Directus quando **ao menos um** destes critérios for atendido:

1. O contrato será consumido por múltiplos clientes (web, mobile, integrações, parceiros).
2. A lógica depende majoritariamente de dados e permissões do próprio Directus.
3. Há evidência de ganho de performance ao centralizar a consulta no backend do Directus.
4. O time aceita manter, versionar e monitorar código de extensão no ecossistema Directus.
5. **A operação de escrita envolve múltiplas tabelas e requer atomicidade** — ou seja, se qualquer etapa falhar, todas as anteriores devem ser desfeitas (rollback). Isso não é possível via chamadas HTTP sequenciais ao Directus; exige acesso direto ao banco via `database.transaction()`.

Se os dados dependem de contexto de aplicação (sessão NextAuth, feature flags, APIs externas), priorize BFF no Next.js.

<Note>
  **Leitura vs. Escrita:** o critério de atomicidade se aplica exclusivamente a operações de **escrita**. Para leitura agregada de múltiplas fontes, o BFF continua sendo o padrão recomendado — a menos que haja outro critério acima satisfeito.
</Note>

### Papel do BFF em operações de escrita (thin proxy)

Quando uma operação de escrita requer endpoint customizado no Directus (ex.: atomicidade), o browser **não chama a extensão diretamente**. O BFF atua como um proxy fino entre o browser e a extensão.

**O BFF em escrita é responsável apenas por:**
1. Extrair o `accessToken` da sessão NextAuth — o browser não pode expô-lo
2. Normalização mínima de campos (ex.: split de nome completo em `first_name`/`last_name`, `email.toLowerCase()`)
3. Repassar a requisição para a extensão Directus com o token no header

**O BFF em escrita NÃO é responsável por:**
- Validação de negócio (fica na extensão Directus)
- Lógica de criação ou atualização (fica na extensão Directus)
- Envio de e-mails ou eventos (fica na extensão Directus)
- Verificação de permissões de dados (fica na extensão Directus via `accountability`)

```
Browser
  └── POST /api/[entidade]        ← rota Next.js (BFF thin proxy)
        └── POST /[entidade]      ← extensão Directus (lógica + trx + e-mail + eventos)
```

Exemplo de BFF thin proxy para criação de talento:

```typescript
// /api/estagio/talent/route.ts
export async function POST(request: Request) {
  const session = await getServerSession(authOptions);
  const accessToken = session?.user?.accessToken; // único motivo de existir

  const body = await request.json();
  const [first_name, ...rest] = body.name.trim().split(" "); // normalização mínima

  const res = await fetch(`${DIRECTUS_URL}/talent`, {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      Authorization: `Bearer ${accessToken}`, // injeta o token
    },
    body: JSON.stringify({ first_name, last_name: rest.join(" "), ...body }),
  });

  const data = await res.json();
  return Response.json(data, { status: res.status });
}
```

<Tip>
  Se você está colocando `if (!email) return 400` ou `await sendEmail()` no BFF de escrita, isso é um sinal de que lógica de negócio está no lugar errado. Mova para a extensão Directus.
</Tip>

### Anti-patterns de performance

Evite os padrões abaixo em listas agregadas:

- **N+1 no BFF**: fazer chamadas por item (ex.: 1 request por talento para a mesma collection).
- **Agregação no cliente**: várias chamadas no browser para montar uma única tabela.
- **Campos excessivos**: usar `fields=*` sem necessidade.

Prefira:

- leitura em lote com filtros `_in`;
- seleção explícita de campos;
- paginação e limites coerentes;
- montagem do DTO no servidor.

### Checklist rápido de decisão

- Quem consome esse contrato (só SSR, cliente web, ou múltiplos sistemas)?
- O dado depende de contexto do app (sessão/permissão/feature flag)?
- A composição está 100% no Directus?
- Existe evidência de gargalo real de performance?
- O contrato precisa ser compartilhado por outros times/sistemas?

<Tip>
  Regra prática: se a necessidade é da aplicação Next, comece com serviço BFF e evolua para endpoint custom no Directus apenas quando houver benefício claro de reúso e performance.
</Tip>

## Convenções de Banco

### Tabelas

| Convenção | Exemplo |
|-----------|---------|
| Plural, snake_case | `talents`, `pulse_responses` |
| Prefixo de módulo | `pulse_`, `match_` |

### Colunas

| Convenção | Exemplo |
|-----------|---------|
| snake_case | `first_name`, `created_at` |
| FKs com `_id` | `tenant_id`, `talent_id` |
| Booleanos com `is_` | `is_active`, `is_anonymous` |
| Timestamps padrão | `created_at`, `updated_at` |

### Índices

Nomenclatura: `idx_{table}_{columns}`

```sql
CREATE INDEX idx_talents_tenant_id ON talents(tenant_id);
CREATE INDEX idx_talents_email ON talents(email);
```

## Convenções de Eventos

### Nomenclatura

Formato: `{domain}.{entity}.{action}`

```typescript
// Exemplos
"pulse.created"
"pulse.response.submitted"
"talent.updated"
"matchmaker.application.status_changed"
```

### Payload

```typescript
interface EventPayload {
  // Identificadores obrigatórios
  tenant_id: string;
  
  // Identificador da entidade
  [entity]_id: string;
  
  // Dados adicionais
  data?: Record<string, unknown>;
  
  // Metadados
  triggered_by?: string;
  timestamp?: string;
}
```
