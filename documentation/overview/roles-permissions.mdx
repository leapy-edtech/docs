## Visão geral

A plataforma Leapy controla o acesso via **roles do Directus** definidos em `leapy-rh`. Cada sessão NextAuth carrega o `role_id` do usuário e o BFF verifica permissões usando a lista `PERMISSIONS` antes de servir dados ou executar ações.

Há dois grupos de roles:

| Grupo | Roles | Contexto |
|-------|-------|----------|
| **Leapy internos** | `system-admin`, `tech-team`, `operations-owner`, `operations-team` | Equipe Leapy — nunca atribuídos a clientes |
| **Clientes** | `rh-account-admin`, `rh-account-manager`, `user-leadership`, `user-learning-leaper`, `intern`, `entidade-account-admin`, `user-candidate` | Usuários das contas cliente |

## Impersonação

A impersonação permite que a equipe Leapy assuma a sessão de um usuário cliente para suporte e diagnóstico. Implementada em `src/lib/auth/permissions/impersonation.ts`.

### Regras

| Regra | Roles |
|-------|-------|
| **Pode iniciar impersonação** | `system-admin`, `tech-team`, `operations-owner` |
| **Protegido — não pode ser impersonado** | `system-admin`, `tech-team` |
| **Targets do botão de impersonação** | `rh-account-admin`, `rh-account-manager`, `user-leadership`, `user-learning-leaper`, `intern` |
| **Timeout de sessão impersonada** | 4 horas (`IMPERSONATION_TIMEOUT_MS = 4 * 60 * 60 * 1000`) |

<Warning>
  `entidade-account-admin`, `operations-team`, `operations-owner` e `user-candidate` **não aparecem no botão de impersonação** — não estão em `IMPERSONATION_TARGET_ROLE_IDS`. Impersonar esses roles requer intervenção direta de `system-admin` ou `tech-team`.
</Warning>

### Fluxo

```
1. Usuário autenticado com role em IMPERSONATION_ALLOWED_ROLE_IDS
2. Clica em "Impersonar" no perfil do usuário target
3. BFF valida: canImpersonate(session.role_id) && canBeImpersonated(target.role_id)
4. Sessão impersonada criada com timestamp (impersonatedAt)
5. Após 4h: isImpersonationExpired(impersonatedAt) = true → sessão encerra
```

## Permissões — constantes

O sistema usa o formato `recurso:ação` definido em `permissions.ts`:

```ts
// Exemplos de permissões geradas
"jovens:manage"         // JOVENS_MANAGE
"pulsos:read"           // PULSOS_READ
"pulsos:create"         // PULSOS_CREATE
"dados_insights:read"   // DADOS_INSIGHTS_READ
"backoffice:access"     // BACKOFFICE_ACCESS
"backoffice:manage"     // BACKOFFICE_IMPERSONATE
```

A verificação no BFF usa `session.user.permissions.includes(PERMISSIONS.JOVENS_MANAGE)` ou equivalente para cada rota protegida.

<Note>
  O recurso `documentos` (`DOCUMENTOS_READ`, `DOCUMENTOS_MANAGE`) existe nas constantes mas está comentado em todos os roles — funcionalidade não ativada.
</Note>
