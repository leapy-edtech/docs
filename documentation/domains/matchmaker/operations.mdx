## Visão Geral

O Matchmaker possui instrumentação completa para monitoramento e diagnóstico via logs estruturados, headers de rastreabilidade e tabela dedicada de search logs.

---

## Logging

### Tabela matchmaker_search_logs

Toda busca (sucesso ou falha) é registrada automaticamente pelo proxy Directus.

| Campo | Descrição | Uso |
|-------|-----------|-----|
| `request_id` | UUID único da requisição | Correlação com logs do Agent System |
| `user_id` | Quem fez a busca | Análise de uso por usuário |
| `query` | Texto da busca | Análise de queries frequentes |
| `success` | Se deu certo | Taxa de sucesso |
| `execution_time_ms` | Tempo total | Performance |
| `candidates_found` | Quantos candidatos | Eficácia |
| `response_data` | Resposta completa (JSONB) | Debug detalhado |

### Consultas Úteis

**Buscas com falha nas últimas 24h:**
```sql
SELECT request_id, user_id, query, error_message, execution_time_ms
FROM matchmaker_search_logs
WHERE success = false
  AND date_created > NOW() - INTERVAL '24 hours'
ORDER BY date_created DESC;
```

**Tempo médio de processamento:**
```sql
SELECT
  DATE_TRUNC('hour', date_created) as hora,
  AVG(execution_time_ms) as tempo_medio,
  COUNT(*) as total_buscas
FROM matchmaker_search_logs
WHERE success = true
  AND date_created > NOW() - INTERVAL '7 days'
GROUP BY hora
ORDER BY hora DESC;
```

**Top queries mais frequentes:**
```sql
SELECT query, COUNT(*) as total, AVG(candidates_found) as media_candidatos
FROM matchmaker_search_logs
WHERE success = true
GROUP BY query
ORDER BY total DESC
LIMIT 20;
```

**Buscas por usuário:**
```sql
SELECT
  du.first_name || ' ' || du.last_name as nome,
  COUNT(*) as total_buscas,
  AVG(msl.execution_time_ms) as tempo_medio,
  AVG(msl.candidates_found) as media_candidatos
FROM matchmaker_search_logs msl
JOIN directus_users du ON du.id = msl.user_id
WHERE msl.date_created > NOW() - INTERVAL '30 days'
GROUP BY du.id, du.first_name, du.last_name
ORDER BY total_buscas DESC;
```

## Headers de Rastreabilidade

Toda resposta do Matchmaker inclui headers para diagnóstico:

| Header | Descrição | Exemplo |
|--------|-----------|---------|
| `X-Request-ID` | UUID único da requisição | `a1b2c3d4-...` |
| `X-Execution-Time` | Tempo em ms (apenas sync) | `2345` |
| `X-RateLimit-Remaining` | Requisições restantes no minuto | `7` |
| `Retry-After` | Segundos para retry (quando 429) | `45` |

## Rate Limiting

### Configuração

| Parâmetro | Valor |
|-----------|-------|
| **Máximo** | 10 requisições |
| **Janela** | 60 segundos (1 minuto) |
| **Key** | `matchmaker:{userId}` |

### Comportamento

1. Cada requisição decrementa o contador
2. Ao exceder: HTTP 429 com `Retry-After`
3. O frontend exibe countdown bloqueando o input
4. Após a janela, o contador reseta

## Timeout

### Configuração

| Componente | Timeout |
|------------|---------|
| **Directus Proxy** | 120.000ms (2 minutos) |
| **Next.js API Route** | 120.000ms (2 minutos) |
| **Agent System** | Sem timeout explícito (depende do proxy) |

### Diagnóstico de Timeout

Se buscas estão dando timeout frequentemente, verificar:

1. **Camada 4 de role** sendo acionada (chamada LLM ~2s)
2. **Muitas skills** para normalizar (>30 skills)
3. **Latência do banco** (verificar conexões do pool)
4. **Latência da OpenAI** (verificar status da API)

## Troubleshooting

### Problema: "Matchmaker não está configurado corretamente"

**Causa**: `AGENT_SYSTEM_URL` não está definida no ambiente do Directus.

**Solução**:
1. Verificar `.env` do Directus
2. Adicionar `AGENT_SYSTEM_URL=https://agent-system-url`
3. Reiniciar o serviço Directus

### Problema: Timeout frequente em buscas com URL

**Causa**: Scraping de sites que bloqueiam ou são lentos.

**Solução**:
1. Verificar se o site está acessível pelo servidor
2. Orientar usuários a colar a descrição manualmente
3. Verificar logs para identificar URLs problemáticas

### Problema: "Nenhuma skill normalizada para buscar candidatos"

**Causa**: Todas as skills extraídas são novas (não existem na base).

**Solução**:
1. Verificar se a base taxonômica está atualizada
2. Verificar se os embeddings estão funcionando (threshold 0.85)
3. Considerar adicionar aliases para skills comuns

### Problema: Poucos candidatos encontrados

**Causa**: Skills muito específicas ou poucos usuários com essas skills no banco.

**Solução**:
1. Verificar o `normalization_result` no log para ver quantas skills foram normalizadas
2. Verificar `flow_decision.strategy` — se é `LLM_MATCH`, o LLM pode estar inferindo skills pouco comuns
3. Verificar a query original — queries muito genéricas podem ter extração pobre

### Problema: Rate limit atingido sem uso excessivo

**Causa**: Múltiplas abas ou sessões do mesmo usuário.

**Solução**: O rate limit é por `userId`, não por sessão. Orientar o usuário a usar apenas uma aba.

## Manutenção

### Limpeza de Logs

Os logs crescem continuamente. Recomenda-se limpar logs antigos periodicamente:

```sql
DELETE FROM matchmaker_search_logs
WHERE date_created < NOW() - INTERVAL '90 days';
```

<Warning>
  Antes de limpar, considere exportar os dados para análise histórica.
</Warning>

### Atualização da Base Taxonômica

A qualidade do matching depende diretamente da base taxonômica. Manter atualizado:

| Ação | Frequência | Impacto |
|------|------------|---------|
| Adicionar role aliases | Sob demanda | Melhora a camada 2 de busca |
| Adicionar skill aliases | Sob demanda | Melhora normalização |
| Revisar skills `pending` | Semanal | Incorpora novas skills ao matching |
| Atualizar embeddings | Mensal | Melhora camada 3 de normalização |
